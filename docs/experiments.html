9. FTS5 Grid Search — Lexical Saturation Test

SQLite FTS5 BM25 (unicode61 tokenizer, 950K chunks) tested as a 5th active RRF arm across 4 weight configurations. Golden v1 (70 queries).

ConfigW_FTS5HitsRecall@20Δ
Baseline (no FTS5)0.03854.3%—

FTS5 medium0.503752.9%−1
FTS5 strong0.753752.9%−1
FTS5 full1.003651.4%−2
Why FTS5 failed: lexical axis saturated
  • TF-IDF arm + Recoll path BM25 already capture all available keyword signal
  • FTS5 adds redundant lexical coverage — same keywords, different scoring
  • Common-keyword noise from FTS5 displaces marginal rank 15–18 correct hits below the top-20 cutoff
  • At W=1.0, two correct hits are pushed from rank ~18–19 to ~21–23

DEAD 0 gains at any weight, 1–2 regressions at W ≥ 0.5. FTS5 weight permanently set to 0.0. Lexical retrieval axis is saturated — further gains require non-lexical approaches (data quality, re-chunking, or entity injection).

Architecture Decision: Dual-Embed

Decision: Add BGE-M3 as 5th RRF arm (do NOT swap Cohere)
  • BGE-M3 must be a retrieval arm (own ANN query → own candidates), not a reranker over Cohere's pool
  • The value is in the union of candidate pools: Cohere-top-N ∪ BGE-top-N ∪ Recoll
  • Separate ChromaDB collection (fulldisk-1k-bge-m3) with identical chunk IDs
  • W_BGE start at 0.5–0.75, grid search with regression-aware objective
  • Target: 83–85/105 = 79–81% Recall@20
  • Cost: ~$0.50–1.00 GCP T4 spot for GPU re-embedding
