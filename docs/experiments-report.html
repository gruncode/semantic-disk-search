Architecture: 5-Arm RRF Pipeline

Reciprocal Rank Fusion (RRF) merges ranked lists from multiple retrieval arms. Score = Σ W/(k+rank) where k=60.

Architecture insight: The 3 active arms cover complementary axes — semantic similarity (BGE vector), exact lexical matching (TF-IDF), and filesystem path/name matching (Recoll path). Adding more lexical arms (Recoll content, FTS5) proved redundant.

Why FTS5 failed — three reasons:
  • Redundant coverage: TF-IDF + Recoll-path already handle keyword matching. Queries FTS5 was supposed to rescue were already top-20 hits.
  • Common-keyword noise: FTS5 matches domain-frequent terms across hundreds of chunks, boosting wrong results' RRF scores and pushing marginal hits (rank 15-18) below the top-20 cutoff.
  • Misleading single-query tests: Individual test queries jumped to rank #1 with FTS5 — but they were already hits in the baseline. Rank improvement ≠ hit conversion.

Key Findings

1. Coverage beats ranking

The reranker experiment proved this definitively: 78% of missed targets aren't in the top-100 candidate pool at all. No amount of reranking or scoring refinement can retrieve a document that isn't in the pool. The retrieval arms' job is coverage — getting the right document into the candidate set.

2. Complementary axes matter, redundant ones don't

The 3 active arms succeed because they cover orthogonal axes: semantic similarity (vector), exact term matching (TF-IDF), and filesystem structure (Recoll path). Adding another lexical arm (FTS5, Recoll content) is redundant — it matches the same terms the existing arms already find, and the noise displaces marginal hits.

3. HyDE is a precision tool, not a general improvement

HyDE helps when the user's query uses different words than the document ("how much is my house worth?" → property valuation report). It hurts when the query already contains the right keywords ("alarm ripple control notes"). This makes it inherently unsuitable as a default — but valuable as an opt-in flag for abstract/reasoning queries.

4. Local models can beat commercial APIs

BGE-M3 (local, free, private) rescued 10 queries that Cohere (cloud API, paid) missed, with zero reverse regressions on the miss set. For domain-specific multilingual content, open-source models can match or exceed commercial embeddings — especially when the domain vocabulary is niche.

5. The remaining gap is data quality, not retrieval

Of the 32 hard misses, most are: scanned PDFs with poor OCR, deep filesystem paths not reflected in chunk text, or Greek semantic gaps in embeddings. No retrieval arm can fix garbled OCR text or missing content. The next improvements must come from upstream: better text extraction, structural re-chunking, or targeted re-OCR of specific documents.

What's Next

Recommended: Stop Adding Arms

Both independent AI advisors agree on this assessment. The retrieval pipeline is at its ceiling. Future work should focus on:

  • Semantic/structural re-chunking (bounded experiment) — the one untested lever that changes what gets embedded upstream of all arms
  • Path/entity candidate injection — deterministic file injection when query terms match filepath tokens, targeting deep-path navigational queries
  • Targeted re-OCR — on-demand for specific documents that matter (legal/financial scans), not a corpus-wide pipeline
  • Curated alias dictionaries — cheap, manual, covers recurring Greek↔English domain terms

What not to pursue: more lexical arms, Recoll content tuning, global HyDE, broad query rewriting, additional rerankers.
